1. Introduction

The Answer Protocol (TAP) is a TCP-based application layer protocol designed for real-time multi-user dungeon (MUD) environments. This protocol enables multiple clients to connect to a shared virtual world where they can interact with the environment, communicate with other players, and participate in collaborative gameplay.

TAP provides a simple, text-based command interface with asynchronous event delivery, making it suitable for both command-line and graphical user interfaces. The protocol emphasizes simplicity, extensibility, and real-time interaction capabilities.

1.1 Requirements Language

The key words "MUST", "MUST NOT", "REQUIRED", "SHALL", "SHALL NOT", "SHOULD", "SHOULD NOT", "RECOMMENDED", "MAY", and "OPTIONAL" in this document are to be interpreted as described in RFC 2119.

1.2 Terminology

  • Client: An application that connects to a TAP server
  • Server: The TAP protocol implementation that manages the virtual world
  • Player: A connected client with an established identity
  • Room: A discrete location within the virtual world
  • Event: An asynchronous message sent from server to client
  • Resource: An item, NPC, or other interactive object in the world

2. Protocol Overview

TAP operates over TCP connections using UTF-8 encoded, line-oriented messages. Each message is terminated by a single newline character (0x0A). The protocol follows a hybrid request-response model with asynchronous event delivery.

2.1 Protocol Characteristics

  • Transport: TCP (Transmission Control Protocol)
  • Encoding: UTF-8
  • Message Delimiter: Line Feed (LF, 0x0A)
  • Connection Model: Persistent connections
  • Message Flow: Bidirectional with asynchronous events

2.2 Protocol State Machine

TAP connections progress through the following states:

  1. DISCONNECTED: No TCP connection established
  2. CONNECTED: TCP connection established, awaiting authentication
  3. AUTHENTICATED: Player identity established, ready for commands
  4. TERMINATED: Connection closed gracefully or due to error

3. Connection Management

3.1 Connection Establishment

Clients MUST establish a TCP connection to the server before sending any TAP messages. Upon successful TCP connection, the client MUST send a CONNECT command to establish player identity.

ABNF Syntax:
connection-sequence = tcp-connect connect-command
tcp-connect = ; Standard TCP three-way handshake
connect-command = "CONNECT" SP username CRLF

3.2 Initial Greeting

Upon successful TCP connection establishment, the server MUST send an initial greeting message to identify the protocol version:

Server → Client: OK hello proto=1

3.3 Authentication Process

After receiving the greeting, the client MUST send a CONNECT command to establish player identity. The server MUST validate that the requested username is unique among active connections.

Server → Client: OK hello proto=1
Client → Server: CONNECT alice
Server → Client: OK connected

If the username is already in use, the server MUST respond with an error:

Client → Server: CONNECT alice
Server → Client: ERR 201 NAME_IN_USE

3.3 Connection Termination

Connections MAY be terminated gracefully using the QUIT command or abruptly due to network conditions. Servers MUST handle both scenarios and clean up player state appropriately.

4. Message Format

4.1 General Message Structure

All TAP messages follow a consistent format:

ABNF Syntax:
message = command-line / response-line / event-line
command-line = command-name [SP arguments] LF
response-line = ("OK" / error-response) [SP response-data] LF
event-line = "EVT" SP event-type SP event-data LF
command-name = 1*ALPHA
arguments = 1*VCHAR
error-response = "ERR" SP error-code SP error-message
error-code = 3DIGIT
event-type = 1*ALPHA
event-data = 1*VCHAR

4.2 Command Messages

Command messages are sent from client to server and consist of a command name followed by optional arguments. Command names are case-insensitive.

4.3 Response Messages

Response messages are sent from server to client in reply to command messages. Responses begin with either "OK" for success or "ERR" for failure.

4.4 Event Messages

Event messages are asynchronous notifications sent from server to client. They begin with "EVT" followed by an event type and associated data.

5. Command Specifications

5.1 Core Commands

5.1.1 CONNECT Command

Syntax: CONNECT

Purpose: Establish player identity for the session

Responses:

  • OK connected - Authentication successful
  • ERR 201 NAME_IN_USE - Username already taken

5.1.2 LOOK Command

Syntax: LOOK

Purpose: Retrieve current room state as JSON

Response: OK

JSON Structure:

{
  "room": {
    "id": "room.identifier",
    "name": "Room Display Name",
    "description": "Room description text",
    "exits": {
      "north": "room.north_id",
      "south": "room.south_id"
    }
  },
  "players": ["username1", "username2"],
  "items": ["item.id1", "item.id2"],
  "npcs": ["npc.id1", "npc.id2"]
}

Example:

Client → Server: LOOK
Server → Client: OK { "room": { "id": "loc.bakery", "name": "Village Bakery", "description": "A warm bakery filled with the scent of fresh bread.", "exits": {"south": "loc.square"} }, "players": ["alice"], "items": ["item.herbs"], "npcs": ["npc.baker"] }

5.1.3 MOVE Command

Syntax: MOVE

Purpose: Move player to adjacent room

Responses:

  • OK room= - Movement successful
  • ERR 301 NO_EXIT - Invalid direction

5.1.4 QUIT Command

Syntax: QUIT

Purpose: Gracefully terminate connection

Response: OK bye

5.2 Communication Commands

5.2.1 CHAT Command

Syntax: CHAT

Purpose: Send chat message to specified scope

Scopes: GLOBAL, ROOM, GROUP

Response: OK

5.2.2 WHO Command

Syntax: WHO

Purpose: Query total player count

Response: OK players=

5.3 Group Management Commands

5.3.1 GROUP CREATE Command

Syntax: GROUP CREATE

Purpose: Create a new player group

Response: OK group=

5.3.2 GROUP INVITE Command

Syntax: GROUP INVITE

Purpose: Invite player to current group

Response: OK

5.3.3 GROUP JOIN Command

Syntax: GROUP JOIN

Purpose: Join an existing group

Response: OK group=

5.3.4 GROUP LEAVE Command

Syntax: GROUP LEAVE

Purpose: Leave current group

Response: OK

5.4 Resource Interaction Commands

5.4.1 TAKE Command

Syntax: TAKE

Purpose: Acquire item from current room

Responses:

  • OK taken= - Item acquired successfully
  • ERR 404 ITEM_NOT_FOUND - Item not available

Example:

Client → Server: LOOK
Server → Client: OK { "room": { ... }, "items":["item.herbs"], ... }
Client → Server: TAKE Herbs
Server → Client: OK taken=item.herbs
Client → Server: LOOK
Server → Client: OK { "room": { ... }, "items":[], ... }

5.4.2 DROP Command

Syntax: DROP

Purpose: Release item to current room

Responses:

  • OK dropped= - Item released successfully
  • ERR 404 ITEM_NOT_IN_INVENTORY - Item not in inventory

Example:

Client → Server: DROP item.herbs
Server → Client: OK dropped=item.herbs
Client → Server: LOOK
Server → Client: OK { "room": { ... }, "items":["item.herbs"], ... }

5.4.3 INVENTORY Command

Syntax: INVENTORY

Purpose: List player's current items as JSON array

Response: OK []

Example:

Client → Server: INVENTORY
Server → Client: OK ["item.herbs", "item.loaf_bread"]

5.4.4 TALK Command

Syntax: TALK

Purpose: Interact with non-player character

Responses:

  • OK - Interaction successful
  • ERR 404 NPC_NOT_FOUND - NPC not present

Example:

Client → Server: TALK Baker
Server → Client: OK Welcome to my bakery!

5.4.5 ATTACK Command

Syntax: ATTACK

Purpose: Initiate combat with an enemy NPC

Responses:

  • OK - Attack successful, returns combat status
  • ERR 404 NPC_NOT_FOUND - NPC not present
  • ERR 405 NPC_NOT_HOSTILE - NPC cannot be attacked

Example:

Client → Server: ATTACK goblin
Server → Client: OK {"attacker_hp": 80, "target_hp": 15, "damage": 10, "status": "combat"}

5.4.6 STATUS Command

Syntax: STATUS

Purpose: Get player's current health and combat status

Response: OK {"hp": , "max_hp": , "status": ""}

Example:

Client → Server: STATUS
Server → Client: OK {"hp": 80, "max_hp": 100, "status": "healthy"}

5.4.7 QUEST Command

Syntax: QUEST

Purpose: Request quest from quest-giver NPC

Responses:

  • OK - Quest available, returns quest information
  • ERR 404 NPC_NOT_FOUND - NPC not present
  • ERR 406 NO_QUEST_AVAILABLE - NPC has no quests or quest already completed

Example:

Client → Server: QUEST merchant
Server → Client: OK {"quest_id": "fetch_herbs", "description": "Bring me 3 healing herbs", "reward": "gold_coin", "status": "available"}

5.4.8 QUESTS Command

Syntax: QUESTS

Purpose: List all player's active and completed quests

Response: OK

Example:

Client → Server: QUESTS
Server → Client: OK [{"quest_id": "fetch_herbs", "status": "active", "progress": "1/3"}, {"quest_id": "defeat_goblin", "status": "completed"}]

6. Implementation Notes

6.1 Work in Progress

IMPORTANT: This RFC intentionally leaves certain aspects underspecified to allow for creative implementation by development teams. The following areas require design decisions and justification:

6.1.1 Combat System Implementation

While basic ATTACK and STATUS commands are defined, the following aspects are left to implementers:

  • Turn-based mechanics: How turns are managed, initiative order, and timing
  • Damage calculation: Formulas for attack damage, defense, and critical hits
  • Combat states: How combat begins, ends, and transitions between states
  • Additional commands: DEFEND, FLEE, USE_ITEM, or other combat-specific actions
  • Status effects: Poison, buffs, debuffs, and their duration management

6.1.2 Quest System Implementation

Basic QUEST and QUESTS commands are provided, but implementers must design:

  • Quest progression: How quest objectives are tracked and validated
  • Quest completion: Automatic vs manual completion mechanisms
  • Quest rewards: Distribution and inventory management
  • Quest dependencies: Prerequisites and quest chains
  • Additional commands: COMPLETE_QUEST, ABANDON_QUEST, or similar

Implementation teams MUST document their design choices and provide justification for their combat and quest system implementations in their project README.

7.1 Event Message Format

Events are asynchronous messages that inform clients of world state changes. All events follow the format:

EVT

Event TypeFormatDescription

8.1 Error Response Format

Error responses follow the format:

7.3 Error Recovery

Clients SHOULD implement appropriate error recovery mechanisms. Non-fatal errors (4xx codes) allow continued operation, while system errors (9xx codes) may require connection re-establishment.

8. Dynamic Resource Management

8.1 Resource Uniqueness

TAP implements a dynamic resource management system where items exist as unique instances within the virtual world. This system prevents resource duplication and maintains world state consistency.

8.2 Item Lifecycle

  1. Spawned: Item exists in a room, available for acquisition
  2. Held: Item is in a player's inventory
  3. Dropped: Item is released back to a room

8.3 Resource Identification

Resources MAY be identified using either:

  • Canonical ID: Unique identifier (e.g., "item.healing_potion")
  • Display Name: Human-readable name (e.g., "Healing Potion")

8.4 Multi-word Resource Names

The protocol supports multi-word resource names. Clients SHOULD handle names containing spaces appropriately in their user interfaces.

9. Security Considerations

9.1 Authentication

TAP provides minimal authentication through username uniqueness. Implementations requiring stronger security SHOULD implement additional authentication mechanisms.

9.2 Network Protocol Handling

TAP implementations MUST correctly handle TCP stream characteristics:

  • Message Fragmentation: Commands may be split across multiple TCP packets. Servers MUST buffer incomplete messages until a complete line is received.
  • Message Coalescing: Multiple commands may arrive in a single TCP packet. Servers MUST process each line-terminated message separately.
  • Encoding: All messages MUST use UTF-8 encoding. Servers MUST handle Unicode characters in usernames and messages without encoding errors.
  • Control Characters: Servers SHOULD either reject or safely handle control characters in messages to prevent terminal manipulation attacks.

9.3 Input Validation

Servers MUST validate all client input to prevent injection attacks and ensure protocol compliance. Malformed messages SHOULD result in appropriate error responses.

9.4 Resource Limits

Implementations SHOULD enforce reasonable limits on:

  • Message length (recommended: 1024 bytes per line)
  • Connection count
  • Resource acquisition per player
  • Chat message frequency
